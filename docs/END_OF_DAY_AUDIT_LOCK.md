# إحصاء نهاية اليوم — التثبيت والقفل (End-of-Day Audit Lock)

> ملاحظة: لم يكن `PROJECT_SUMMARY.md` ولا `AUDIT_REPORT.md` موجودَين في المستودع وقت هذا
> التغيير، فوُثِّق التصميم هنا في ملف مخصّص. عند إنشاء `PROJECT_SUMMARY.md` لاحقاً يُنقَل هذا
> القسم إليه.

## الفكرة
كان "المبلغ الفعلي" حقلاً إجمالياً واحداً قابلاً للتعديل في أي وقت — فلا فرق فعلي بين
"مسودة قيد الإدخال" و"إحصاء مُدقَّق مُثبَّت". أصبح الآن:

1. **ثلاث خانات فعلية حسب طريقة الدفع** (كاش/فيزا/شيك) بدل حقل إجمالي واحد.
2. **النظام محسوب لكل طريقة** عبر `cashAudit:getSystemBreakdown` (موجود مسبقاً): صافي
   الحركة لكل طريقة = وارد (دفعات العملاء + سداد ديونهم) − صادر (مدفوعات الموردين).
   > **افتراض موثّق:** `daily_expenses` و`salary_payments` **لا تملكان عمود `method`**،
   > فتُحتسَب دائماً **كاش** لغرض هذا الإحصاء فقط. قرار منطقي (لا حقيقة موثّقة سابقاً).
   > اختبار تحقّق: `cash + visa + cheque` يجب أن يساوي `report:daily.net` بالضبط.
   > **فيزا في الصادر:** تُتتبَّع لمدفوعات الموردين بالفيزا (لها عمود `method`)؛ الفيزا
   > الواردة عادةً تُحصَّل بنكياً، لكن الخانة اليدوية أُبقيت كما طُلب لتغطية أي عدّ يدوي.

## زر «تثبيت وقفل الإحصاء»
تسمية مقصودة: هو **ليس مجرد حفظ** بل **قفل فعلي**. بعد الضغط (وتأكيد التنبيه):
- تُحفَظ القيم الثلاث الفعلية + لقطة النظام لكل طريقة + الفروق، ويُضبط `is_locked=1`.
- تختفي صناديق الإدخال تماماً وتُستبدَل بنص عادي لكل رقم (يحلّ نهائياً مشكلة "عرض
  الصندوق أضيق من الرقم"، إذ لم يعد ثمة صندوق محدود العرض بعد القفل).
- بجانب كل رقم زر **«تعديل» مستقل** لذلك الرقم وحده.

## تعديل رقم مُثبَّت (المسار الوحيد المسموح)
- زر «تعديل» يمرّ عبر `ConfirmDialog` **بكلمة السر** (سجل مالي مُدقَّق).
- بعد التأكيد يُعاد فتح **ذلك الحقل وحده** كصندوق إدخال + «حفظ التعديل»، وباقي الأرقام
  تبقى نصاً مقفلاً.
- عند الحفظ: **النظام يبقى مجمَّداً** كما وقت التثبيت؛ **الفرق يُعاد حسابه فقط**.
- يُسجَّل كل تعديل في `activity_log` (`entity_type='cash_audit'`, `action_type='update'`).
- التثبيت الأول يُسجَّل بـ `action_type='lock'`.

## القفل مفروض في الـ backend (وليس الواجهة فقط)
- `cashAudit:save` **يرفض** أي كتابة فوق سجل `is_locked=1` مهما كان المسار.
- المسار الوحيد للكتابة فوق سجل مقفل هو `cashAudit:updateLocked`، الذي **يُعيد التحقق من
  كلمة السر داخل الـ main process** (`verifyPassword`) — فلا يمكن تعديل سجل مُدقَّق حتى
  باستدعاء القناة مباشرةً متجاوزاً الواجهة. (`ConfirmDialog.onConfirm` يمرّر كلمة السر
  المُتحقَّقة إلى المستدعي ليعيد تمريرها للـ backend.)

## أعمدة `daily_cash_audits` الجديدة
| عمود | معنى |
|------|------|
| `actual_cash` / `actual_visa` / `actual_check` | الفعلي المُدخل يدوياً لكل طريقة |
| `system_cash` / `system_visa` / `system_check` | لقطة النظام لكل طريقة وقت التثبيت |
| `is_locked` | 0 = مسودة، 1 = مُثبَّت ومقفل |

> **تسمية:** استُخدم `system_check` (لا `system_cheque`) توافقاً مع العمود الموجود
> مسبقاً `actual_check`، تفادياً لخلط تسميتين لنفس المفهوم في الجدول نفسه.

الأعمدة القديمة `system_total` / `actual_amount` / `difference` **أُبقيت** للتوافق مع أي
تقرير/طباعة قديمة، وتبقى دائماً = مجموع الأعمدة المفصّلة المقابلة (الفرق = الفعلي − النظام).

### الترحيل (migration)
`migrateCashAuditLock` في `src/database.ts` (بأسلوب `migrateChequeStatus`):
- يضيف الأعمدة الأربعة الجديدة عبر `ALTER TABLE ... ADD COLUMN`.
- **عند أول إضافة لعمود `is_locked` فقط:** يُقفَل كل سجل موجود (`is_locked=1`) لأنه كان
  "مُثبَّتاً" حسب المنطق القديم. للسجلات بلا تفصيل حسب الطريقة (الأعمدة المفصّلة أصفار):
  `actual_cash = actual_amount` و`system_cash = system_total` (أفضل افتراض ممكن — التفصيل
  لم يكن موجوداً وقت تسجيلها، ونفترض أن الكل كان كاشاً).
