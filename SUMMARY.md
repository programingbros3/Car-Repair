# ملخّص المشروع الكامل — كراج الخط الأخضر (Green Line Garage)

> نظام إدارة كراج شامل يعمل كتطبيق سطح مكتب (Desktop) على ويندوز، يدير الصيانة والبيع المباشر
> والموردين والعملاء والكفالات والفوترة والصندوق والتقارير المالية — بواجهة عربية RTL بالكامل.

---

## 1. نظرة عامة

| البند | القيمة |
|---|---|
| **الاسم البرمجي** | `car-repair-shop` |
| **الإصدار** | `1.0.0` |
| **الوصف** | نظام إدارة كراج شامل: إدارة الصيانة، البيع المباشر، سجلات العملاء، الكفالات، الفوترة، والتقارير المالية |
| **المطوّرون** | Anas Jaidi، Hashem Khader، Muhammad Mansour |
| **النوع** | تطبيق Electron محلي (Offline-first)، بلا خادم ولا اتصال سحابي مباشر |
| **اللغة/الاتجاه** | عربي بالكامل، RTL (`direction: rtl`) |
| **العملة** | الشيكل (₪) |
| **المنصّة الأساسية** | Windows 11 (x64) — مع دعم بناء لـ macOS و Linux في الإعدادات |

المشروع موجّه لكراج سيارات فعلي: صاحب المحل يفتح التطبيق يومياً لتسجيل عمليات الصيانة، البيع المباشر
للقطع، فواتير الموردين، المصاريف، الرواتب، ومتابعة الصندوق والديون والشيكات والكفالات وتقارير الأرباح.

## 2. البنية التقنية (Tech Stack)

### الطبقة الأساسية
- **Electron ^30** — غلاف سطح المكتب (main + preload + renderer).
- **React ^18.2** + **TypeScript ^5.2** — واجهة المستخدم.
- **Vite ^5.1** + `vite-plugin-electron` — أداة البناء والتطوير.
- **better-sqlite3 ^12.11** — قاعدة بيانات SQLite محلية متزامنة (synchronous) في الـ main process.
- **react-router-dom ^7** — التوجيه (باستخدام `HashRouter` لملاءمة `file://`).

### مكتبات مساعدة
- **bcryptjs** — تجزئة كلمة سر التطبيق (hash).
- **node-machine-id** — قراءة معرّف الجهاز الفريد (HWID) لقفل الترخيص.
- **fuse.js** — بحث ضبابي (fuzzy search) في الواجهة.
- **xlsx (SheetJS)** — تصدير Excel.
- **@fontsource/tajawal** — الخط العربي (Tajawal).

### أدوات التطوير
- **electron-builder ^24** — تحزيم النسخة النهائية (NSIS installer / portable).
- **@electron/rebuild** — إعادة بناء الـ ABI الأصلي لـ better-sqlite3 بين Node و Electron.
- **ESLint** + `@typescript-eslint` + إضافات React hooks/refresh.

### أوامر npm الرئيسية
```
dev               → vite (تطوير مع Hot Reload، القفل الأمني معطّل)
typecheck         → tsc --noEmit
build             → tsc && vite build && electron-builder
build:win         → tsc && vite build && node scripts/pack-win.mjs
lint              → eslint (max-warnings 0)
reset-dev-license → مسح ترخيص جهاز التطوير (PowerShell)
verify:integrity  → تشغيل فحوص سلامة قاعدة البيانات (scripts/run-verify.mjs)
test:bulk         → اختبار الدفعات العامة للموردين
```

## 3. المعمارية الطبقية (Architecture)

التطبيق مبني على **فصل صارم للطبقات** مع عقد واضح بين كل طبقة والتي تليها:

```
┌─────────────────────────────────────────────────────────────┐
│  React UI (src/pages, src/components)  — camelCase           │
│  ▲ تستهلك البيانات عبر GarageContext فقط                     │
├─────────────────────────────────────────────────────────────┤
│  GarageContext (src/store) — حالة عامة + تحميل كسول lazy      │
│  ▲ يستدعي dbService حصراً                                    │
├─────────────────────────────────────────────────────────────┤
│  services/db.ts — الواجهة الوحيدة بين React والـ IPC          │
│  ▲ يحوّل camelCase ⇄ snake_case عبر utils/dbMapper           │
│  ▲ يفكّ غلاف { success, data, error }                        │
├─────────────────────────────────────────────────────────────┤
│  preload.ts — جسر contextBridge + ALLOWED_CHANNELS whitelist │
│  ══════════════ حدود العملية (Process Boundary) ═════════════│
│  ipc-handlers.ts — كل معالجات IPC (Layer الربط)             │
│  ▲ يستدعي منطق الأعمال + SQL مضمّن للفجوات                    │
├─────────────────────────────────────────────────────────────┤
│  src/db/* (Layer A) — منطق الأعمال الأساسي (snake_case)      │
│  electron/*.ts — ميزات مستقلة (auth, license, backup, vat)   │
├─────────────────────────────────────────────────────────────┤
│  better-sqlite3 → garage.db (في مجلد userData)              │
└─────────────────────────────────────────────────────────────┘
```

### القواعد الحاكمة (Contracts)
1. **snake_case على السلك، camelCase في الواجهة.** التحويل يتمّ حصراً في `src/utils/dbMapper.ts` عبر `services/db.ts`.
2. **لا استدعاء مباشر لـ `window.ipcRenderer`** من الصفحات أو الـ Context — كل شيء يمرّ عبر `dbService`.
3. **كل معالج IPC** ملفوف بـ `on()` ويُعيد `{ success, data }` أو `{ success: false, error }`.
4. **منطق الأعمال يعيش في `src/db/*`** (مشترك بين الـ main process ونصوص المحاكاة)؛ ملف `ipc-handlers.ts` يربط القنوات فقط مع SQL مضمّن للفجوات (الحذف/التعديل/الدليل/الكفالات/العروض المجمّعة).
5. **ميزات الأمان/النسخ منفصلة تماماً** كل واحدة في ملفها (`auth.ts`, `license.ts`, `auto-backup.ts`, `vat.ts`)، تأخذ `db` كمعامل، وتخزّن حالتها في جدول `app_settings` (key/value).

### الأمان على مستوى العملية (Process Security)
- `preload.ts` **لا يعرّض `ipcRenderer` كاملاً**؛ فقط دالة `invoke` مقيّدة بقائمة بيضاء `ALLOWED_CHANNELS`. أي قناة خارجها تُرفَض قبل الوصول للـ main process.
- في النسخة النهائية (exe): تُعطَّل أدوات المطوّر (`devTools: false`) وتُزال قائمة التطبيق (`Menu.setApplicationMenu(null)`) — لمنع فتح DevTools بأي اختصار وكشف الكود.
- روابط `shell:openExternal` مقيّدة على `http/https` فقط (منع `file:`/`javascript:`).

## 5. قاعدة البيانات (SQLite Schema)

`PRAGMA journal_mode = WAL` + `PRAGMA foreign_keys = ON`. القاعدة في `userData/garage.db`
(أو `GARAGE_DB_PATH` للمعاينة المعزولة). الطوابع الزمنية بتوقيت محلي `datetime('now','localtime')`.

### الجداول الرئيسية

| الجدول | الغرض |
|---|---|
| `maintenance_invoices` | فواتير الصيانة (زبون، سيارة، حالة `in_progress`/`delivered`، خصم، إجمالي/مدفوع/متبقّي) |
| `direct_sale_invoices` | فواتير البيع المباشر للقطع |
| `invoice_items` | بنود الفاتورة (صيانة أو بيع مباشر) — يميّزها `invoice_type`؛ للصيانة: `part_type` (part/service)، `customer_owned`، `warranty` للبند |
| `payments` | دفعات الاستلام/التسليم (عند إنشاء الفاتورة أو تسليم السيارة) |
| `debt_payments` | دفعات تحصيل الدين لاحقاً (من شاشة الديون المعلّقة) |
| `payment_cheque` / `payment_visa` | تفاصيل الشيك/الفيزا لدفعات العملاء |
| `debt_payment_cheque` / `debt_payment_visa` | تفاصيل شيك/فيزا سداد الديون |
| `suppliers` | دليل الموردين المتكرّرين |
| `supplier_invoices` | فواتير الموردين (شراء) |
| `supplier_items` | بنود فاتورة المورد (بخصم على مستوى البند) |
| `supplier_payments` / `supplier_debt_payments` | دفع/سداد دين المورد (+ جداول cheque/visa لكل منها) |
| `supplier_bulk_payments` | **دفعة عامة**: مبلغ واحد يُوزَّع على عدة فواتير مورد |
| `supplier_bulk_payment_allocations` | توزيع الدفعة العامة على الفواتير (يشير لصف `supplier_debt_payments` الفعلي) |
| `daily_expenses` | المصاريف اليومية |
| `employees` | الموظفون (بأجر يومي) |
| `salary_payments` | دفعات الرواتب (لقطة الأجر + أيام العمل + مكافأة/خصم) |
| `cash_ledger` | **الصندوق**: كل حركة نقدية (وارد/صادر) مع رصيد تراكمي `balance_after` وطريقة الدفع |
| `warranties` | الكفالات (مصدرها صيانة أو بيع مباشر، مع مدة period_value/period_unit) |
| `daily_cash_audits` | إحصاء نهاية اليوم (نظام مقابل فعلي، مقسّم كاش/فيزا/شيك، قفل `is_locked`) |
| `app_settings` | إعدادات key/value (كلمة السر، النسخ الاحتياطي، الترخيص، القفل، الضريبة) |
| `activity_log` | سجل النشاط الخفيف (تعديل/حذف/تسليم/قفل) للمراجعة |

### فهارس (Indexes)
فهارس على النمرة/الحالة/التاريخ للصيانة، على `invoice_id+invoice_type` للبنود والدفعات، على تاريخ الصندوق ومرجعه، على حالة الشيكات، وفهرس **فريد** على رقم الفاتورة لكل جدول (شرطي على `IS NOT NULL`).

### الترحيل (Migrations) — `src/database.ts`
`applySchemaAndMigrations()` تُنفَّذ في **كل إقلاع** وهي **idempotent** بالكامل:
1. `db.exec(schema)` — كل `CREATE TABLE IF NOT EXISTS` والفهارس (تعمل على القواعد القديمة أيضاً).
2. حلقة `ALTER TABLE ADD COLUMN` — تتخطّى `duplicate column name` بأمان (أرقام الفواتير، الخصومات، خصم التسوية، `method` في الصندوق، تفاصيل الكفالة/الأجر…).
3. ترحيلات خاصّة:
   - `migrateChequeStatus` — إضافة حالة الشيك (pending/cashed/bounced) + الشيكات القديمة تُعتبر `cashed` مرة واحدة.
   - `migrateLedgerMethod` — ملء `method` للصفوف القديمة من نمط الملاحظة.
   - `migrateCashAuditLock` — تفصيل الإحصاء حسب طريقة الدفع + قفل السجلات القديمة.
   - `migrateUniqueInvoiceNumbers` — قيود التفرّد على أرقام الفواتير.
   - `backfillInvoiceNumbers` — تعبئة أرقام الفواتير القديمة بصيغة `{prefix}-{year}-{seq}`.

نفس الدالة مُصدَّرة ويستخدمها مسار **الاستيراد** لترقية أي نسخة احتياطية قديمة قبل تفعيلها (بلا تكرار منطق).

## 6. الوحدات المنطقية (Business Logic) بالتفصيل

### الصندوق (`ledger.ts`) — قلب النظام المالي
- `cash_ledger` هو **المصدر الوحيد للحقيقة** لكل الحركة النقدية.
- `recordLedgerEntry()` — يضيف قيداً محسوباً رصيده التراكمي حسب **الترتيب الزمني** لا ترتيب الإدخال. القيد بتاريخ سابق (backdated) يُطلق `recomputeLedgerBalances()` لإعادة حساب كل الأرصدة.
- `REF` — أنواع مراجع ثابتة (maintenance_payment, maintenance_release, direct_sale_payment, debt_customer, supplier_payment, supplier_debt, daily_expense, salary) تربط كل قيد بمصدره.

### الدفعات وخصم التسوية (`payments.ts`)
- `addPayment` / `addDebtPayment` — يتحقّقان من عدم تجاوز المتبقّي، يدرجان الدفعة، يحدّثان `amount_paid/amount_remaining`، ويسجّلان قيد صندوق.
- **خصم التسوية (`settlement_discount`)** — يُسقِط مبلغاً من المتبقّي **دون** تسجيله كنقدية في الصندوق. يُدرَج كصف دفعة `amount=0` مع توثيق، فيبقى الثابت: `total = paid + remaining + settlement`.

### طريقة الشيك (M3) — دلالة مهمّة
- **الشيك لا يدخل الصندوق نقداً إلا عند صرفه فعلياً.** عند الاستلام يُحتسب مدفوعاً على الفاتورة فقط.
- `updateChequeStatus` (`cheques.ts`) يطبّق أي انتقال بين الحالات كفرق (delta) على بُعدين: قيد الصندوق ومدفوع الفاتورة:
  - `pending` → محتسَب على الفاتورة، بلا صندوق.
  - `cashed` → محتسَب + قيد صندوق (بتاريخ الصرف).
  - `bounced` → غير محتسَب (يعود الدين)، بلا صندوق.

### الصيانة (`maintenance.ts`)
- إنشاء الفاتورة، تحديثها (استبدال البنود + إعادة حساب الإجمالي/المتبقّي)، **تسليم السيارة** (`releaseMaintenanceCar`) بدفعة إضافية وتغيير الحالة، وتاريخ السيارة حسب النمرة.
- البنود تدعم `customer_owned` (قطعة الزبون لا تُحسب في الإجمالي)، `part_type` (قطعة/خدمة)، وكفالة لكل بند.

### البيع المباشر (`direct-sale.ts`)
- مشابه للصيانة بلا سيارة؛ خصم على مستوى الفاتورة؛ كفالة شاملة واحدة.

### الموردون والدفعات العامة (`suppliers.ts`)
- فواتير المورد ببنود لها خصم على مستوى البند، دفع وسداد دين، ودليل موردين.
- **الدفعة العامة (Bulk Payment)** — ابتكار مهمّ: مبلغ واحد (بطريقة دفع واحدة، حتى شيك واحد) يُوزَّع على عدة فواتير غير مسدَّدة. كل توزيع يُسجَّل كصف `supplier_debt_payments` عادي (فتبقى دورة الصندوق والشيكات وحذف الفواتير بلا تغيير)، مع ترويسة تجمعها في `supplier_bulk_payments`. يتحقّق النظام أن مجموع التوزيع = مبلغ الدفعة، وأنه لا يتجاوز إجمالي ديون المورد.

### الكفالات (`warranties`)
- تُزامَن تلقائياً من الفواتير عبر `reconcileWarranties` (مطابقة تفاضلية: تحديث الموجود في مكانه، إدراج الجديد، حذف المُزال) — **مع الحفاظ على حقل `notes` المُحرَّر يدوياً** عند إعادة حفظ الفاتورة الأم.
- `calcEndDate` (`utils/warranty.ts`) يحسب تاريخ الانتهاء بحساب UTC صرف مع تثبيت آخر يوم في الشهر عند التجاوز (مثال: 31 يناير + شهر = 28 فبراير).

### التقارير (`reports.ts`)
- **يومي** — تفصيل الوارد/الصادر لليوم من الصندوق.
- **شهري** — إجماليات + تفصيل حسب النوع + عدّ الفواتير + الديون الجديدة + الخصومات الممنوحة + خصومات التسوية + عدد الكفالات.
- **الديون** — عملاء + موردون بإجماليات.
- **أعمار الديون** — تصنيف (0-30 / 31-60 / 61-90 / 90+).
- **أفضل العملاء** — تجميع بمفتاح الهاتف المُنظَّف (نفس الزبون بصيغ هاتف مختلفة يُدمَج).

### ترقيم الفواتير (`invoiceNumber.ts`)
- صيغة `{prefix}-{year}-{seq}` بأربع خانات تُعاد للصفر كل سنة.
- **الصيانة والبيع المباشر يشتركان بتسلسل واحد (`INV`)** لأنهما يُعرضان كفاتورة بيع موحّدة؛ الموردون مستقلون (`PUR`).

## 7. الأمان والترخيص

### كلمة السر (`auth.ts`)
- تُخزَّن كـ **bcrypt hash** (10 جولات) في `app_settings` — **لا كلمة سر افتراضية**.
- أول تشغيل بلا hash ⇒ شاشة "تعيين كلمة السر" (حد أدنى 6 أحرف).
- **التحقق يتمّ في الـ main process فقط** — لا يمكن كشفها من DevTools.
- **قفل تصاعدي عند تجاوز المحاولات**: 5 محاولات ⇒ قفل 30 ثانية → دقيقة → 5 دقائق.
- كلمة السر **تتبع الجهاز لا الملف المستورد** — عند الاستيراد تُطبَّق حالة مصادقة الجهاز الحالية على النسخة المستوردة.

### القفل التلقائي (`useAutoLock.ts`)
- يقفل التطبيق عند الخمول (افتراضياً 10 دقائق، قابل للتعديل/التعطيل).

### قفل الجهاز — الترخيص (`license.ts`)
- عند أول تشغيل: يأخذ HWID الجهاز، يوقّعه بـ **HMAC-SHA256** (سرّ يُحقَن وقت البناء عبر `vite define` من `GARAGE_LICENSE_SECRET` أو `electron/.license-secret`)، ويخزّنه في القاعدة.
- كل إقلاع لاحق: يتحقّق أن الجهاز الحالي = المربوط. النسخ لجهاز آخر ⇒ HWID مختلف ⇒ **حظر**.
- **حماية ضد إعادة الربط بحذف الصفوف**: علامة خارجية تُكتب في مكانين خارج القاعدة (سجل ويندوز Registry + ملف مخفي `.glg-lic` في userData). إن اختفت صفوف الترخيص لكن العلامة موجودة لنفس الجهاز ⇒ تعافٍ؛ لجهاز آخر ⇒ حظر.
- **Fail-open** عند تعذّر قراءة HWID (تعطيل عميل شرعي أسوأ من تمرير ناسخ محتمل) — يستمر التطبيق مع تحذير.
- القفل **معطّل تماماً في وضع التطوير** (`npm run dev`)، يعمل فقط في النسخة النهائية.

### سجل النشاط (`activity_log`)
- توثيق خفيف لكل عملية حسّاسة (تعديل/حذف/تسليم/قفل/دفعة عامة) بوصف عربي مقروء.

### ترجمة الأخطاء (`toArabicError`)
- أخطاء SQLite الخام (FOREIGN KEY / UNIQUE / NOT NULL / no such table) تُترجَم لرسائل عربية واضحة للمستخدم.

## 8. النسخ الاحتياطي

### يدوي (`ipc-handlers.ts`)
- **تصدير** — `wal_checkpoint(FULL)` ثم نسخ `garage.db` لملف يختاره المستخدم.
- **استيراد** — أسلوب آمن: يُحضّر النسخة على ملف مؤقّت (ترقية بنيتها + تثبيت كلمة سر الجهاز) و**يبدّل قاعدة الإنتاج فقط عند النجاح** (مع نسخة أمان). النسخة التالفة/القديمة تفشل بلا مساس بالإنتاج، ثم إعادة تشغيل.

### تلقائي (`auto-backup.ts`) — منفصل تماماً
- نسخ دوري إلى مجلد يحدّده المستخدم (عادة مجلد مزامنة سحابية محلي مثل Google Drive Desktop) — التطبيق لا يتعامل مع أي API سحابي.
- اسم الملف بطابع زمني، **rotation** يحتفظ بعدد محدّد (افتراضياً 14).
- يُشغَّل عند الإقلاع (إن مرّ يوم) وعند الإغلاق.
- **وجهة ثانية اختيارية** (مجلد سحابي ثانٍ) بنسخ مستقل — فشلها لا يُفشل الأساسية.
- لا يرمي استثناءً أبداً (لا يوقف التطبيق).

## 12. مبادئ التصميم الحاكمة (خلاصة فلسفية)

1. **مصدر واحد للحقيقة**: الصندوق (`cash_ledger`) لكل حركة نقدية؛ `amount_paid/remaining` على الفواتير مشتقّة ومتّسقة مع الثابت `total = paid + remaining + settlement`.
2. **الترحيلات idempotent** وتعمل على القواعد القديمة والنسخ المستوردة معاً — "لا نحذف بيانات قديمة".
3. **فصل الطبقات الصارم** مع عقود واضحة (snake_case ⇄ camelCase، `{success,data,error}`، قائمة قنوات بيضاء).
4. **الأمان دفاعي متعدّد الطبقات**: تحقّق في الـ backend لا الواجهة فقط، قفل جهاز بعلامة خارجية، كلمة سر hash، قائمة قنوات بيضاء.
5. **الميزات المستقلّة معزولة** (auth/license/backup/vat) كل واحدة بملفها وحالتها في `app_settings`.
6. **الأداء مع البيانات الضخمة**: تحميل كسول، إعادة جلب مُوجّهة بالنطاقات، فهرسة مدروسة.
7. **تجربة عربية أصيلة**: RTL كامل، رسائل أخطاء عربية، تواريخ عربية، عملة الشيكل.
